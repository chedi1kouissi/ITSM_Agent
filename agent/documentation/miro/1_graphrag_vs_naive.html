Architectural Choice: Why GraphRAG?

Comparing a standard LLM agent against our context-aware topological agent

⚠️ Standard Agent (Logs Only)

Log: "payment-api: context deadline exceeded"
Log: "db-node: max connections reached"
  • Blind Fixes: The agent sees the API failing and suggests restarting the payment-api. It doesn't know the DB is shared.
  • Missed Blast Radius: The agent has no idea that catalog-api also relies on the same overloaded database.
  • Hallucinations: It proposes scaling up the app tier, which actually makes the DB connection exhaustion worse.
  • No Infrastructure Context: It misses the API Gateway timeout settings sitting in front of the services.

✨ Our Agent (Neo4j GraphRAG)

Agent automatically queries Neo4j for the topological state alongside the logs.
  • Deterministic Context: Instead of guessing, the agent queries the Graph. It maps exactly which microservices depend on the failing DB.
  • Cross-App Awareness (Blast Radius): The graph proves that multiple distinct apps map to the same physical host (postgres-cluster-prod).
  • Targeted Recoveries: Because the graph stores properties, the LLM knows the exact pool_size and timeout_ms variables to suggest changing.
  • Safety First: The agent understands exactly what it's breaking if it restarts a critical upstream node.

🛠️ The GraphRAG Toolchain (FastMCP)

The LLM accesses the Neo4j topology deterministically using three structured workflow tools:

get_service_dependencies()
Fetches a 2-hop forward map of the application architecture. Pulls in node criticalities [P0, P1] and edge configurations (latency SLOs, ORM limits, use-cases).
get_blast_radius()
Reverse cross-app lookup. When a shared database or cache is failing, this tool queries the graph backward to find every distinct service that will be impacted.
get_infrastructure_routes()
Maps the frontend infrastructure. Identifies which API Gateways or Load Balancers are routing traffic to the failing services, grabbing their specific protocol timeouts.
