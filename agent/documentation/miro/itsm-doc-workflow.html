📥 Receive Log Batch Input
Agent receives a JSON payload containing categorized logs: app_logs, database_logs, infrastructure_logs, and monitoring_logs. The system prompt activates the SRE persona and injects the mandatory workflow sequence. Raw logs are immediately archived to the database for audit and reproducibility.

02
🎯 Initialize Incident Record Tool Call #1
The first mandatory tool call. Agent creates the incident record in PostgreSQL with status ANALYZING. The incident_id and app_id are assigned here and must be reused in all subsequent tool calls. No other tool can be called before this one.
initialize_incident() incidents table → ANALYZING status agent_actions table

03
🔍 Collect & Save Evidence Analysis
Agent analyzes all log categories, cross-referencing timestamps to identify the causal chain (e.g., slow query → pool exhaustion → OOMKilled). Each relevant log line is saved with its source, timestamp, and reasoning — explaining why it is proof of the root cause. This builds a queryable, structured evidence chain in the database.
add_evidence() log_agent_action() (optional) evidence table
Source app / db / infra / monitoring
Output Ordered causal chain
Rule BEFORE finalize_incident
04
🔧 Create & Score Recovery Plan Remediation
Based on evidence, the agent formulates a step-by-step recovery plan. Each step is saved individually with order, description, and risk level (HIGH / MEDIUM / LOW). The full plan text is then passed to calculate_risk_score() which detects dangerous keywords (e.g., drop, rm -rf, delete all) and returns a score from 0 to 100.
add_recovery_steps() calculate_risk_score() recovery_steps table
Risk Score 0–100 (keyword-based)
Levels HIGH / MEDIUM / LOW per step

✅ Finalize Incident Ticket Output · LAST STEP
The final mandatory tool call. Status changes from ANALYZING → OPEN. The ticket is populated with the full recovery plan, risk score, and complete agent analysis notes. The incident is now visible to the ITSM team and ready for tracking, execution, and SRE review.
