Long-Term Memory (LTM) Architecture

Why we chose graph-embedded vector storage over a standalone vector database for autonomous incident resolution.

⚠️ The Standard Approach (Standalone Vector DB)

1. LLM detects "Timeout Error"
↓ (Semantic Search)
Chroma / Pinecone
↓ (Results retrieved)
Returns 5 random tickets containing the word "Timeout" from entirely unrelated microservices.

Most AI agents use a completely separate Vector DB for their "memory".

The fatal flaw: Vector math looks for semantic similarity (text), not topological relevance. Therefore, if the payment-db crashes, the Vector DB might retrieve an old ticket about the frontend-ui timing out just because the text strings mathematical match. The agent hallucinates a fix for the wrong service.

🧠 Our Approach (Neo4j Vector Indices)

1. Graph identifies affected node: payment-db
↓ (Localized Vector Search)
Neo4j Node: (payment-db)
🎫 Vectorized Ticket #121 🎫 Vectorized Ticket #184
↓ (100% Contextual Match)
Returns specific historical fixes applied only to this specific database container.

We embed the incident resolution text into a dense vector... and store it directly as a property or attached vertex on the problematic topological node.

Why this dominates standard architectures:

We eliminate the "Search space hallucination". When the agent interrogates the system, it doesn't search a massive global pool of historical text. It navigates the graph to the failing node, pulls only the memory attached to that precise component, and applies the exact historical fix.
