GraphRAG Architecture for the ITSM Agent
Implementing GraphRAG (Graph-based Retrieval-Augmented Generation) for your ITSM Agent will turn it into a highly contextual, self-learning system. Here is a breakdown of how it would work across multiple apps and how it continuously learns from human feedback.

1. Multi-App Architecture (The "Regions" Approach)
You are exactly right—you would structure the Knowledge Graph with separate "regions" or "subgraphs" for each application, connected by shared infrastructure.

The Graph Structure (Nodes & Edges)
App Nodes: Represent the top-level application (e.g., App 04).
Service Nodes: Represent the microservices inside the app (e.g., payment-api, auth-service).
Infrastructure Nodes: Represent shared resources (e.g., postgres-cluster-prod, api-gateway).
Relationships (Edges): DEPENDS_ON, HOSTED_ON, CALLS, BELONGS_TO.
How it looks in the Database
Even if you use PostgreSQL instead of Neo4j, it looks like this: If App 04 (payment-api) queries a central database, and App 05 (billing-api) queries the same database, the graph links them.

Why this is powerful: If payment-api goes down, the agent queries the graph and sees it depends on postgres-cluster-prod. It checks the graph again and sees billing-api also depends on postgres-cluster-prod. If both are showing errors, the agent instantly deduces the database is the root cause, not the individual apps!
2. Attaching "Vector Memory" to Graph Nodes (The RAG Part)
A Knowledge Graph tells the agent what the architecture is. The Vector Database tells the agent how to fix it.

Instead of throwing all your past tickets into one giant vector pool, you attach the vectors (embeddings) directly to the specific nodes in the graph.

Node: payment-api
Attached Vector Memory:
Past Incident A: "Jan 12 Timeout -> Fixed by scaling up the pod replicas."
Past Incident B: "Feb 05 Timeout -> Fixed by resetting the stripe gateway key."
Runbook Component: "Standard procedure for payment-api restarts."
When the agent investigates payment-api, it only searches the vectors attached to the payment-api node (and its direct dependencies). This eliminates hallucination where the LLM might suggest a fix meant for a totally different app.

3. The Continuous Feedback Loop (How it learns)
This is the most critical part of an enterprise AI: Long Term Memory (LTM). How does the graph get smarter over time?

Step A: The Human Rejection Loop
The Triage Agent creates a ticket: "Root Cause: DB Connection Timeout. Remediation: Restart the DB."
The High-Risk ticket goes to the Human SRE.
The Human SRE rejects the plan and types: "No, never restart the DB for this error code. The correct fix is to flush the Redis cache first."
Step B: The Knowledge Extractor (Triggered via API)
When a ticket is successfully closed (whether by the AI or corrected by a human), the final, approved resolution is sent to a background script.
An LLM processes the closed ticket to extract the "Lesson Learned."
It generates a clean text summary: "Error pattern [X] on service [Y] is resolved by action [Z]. Context: [Human SRE Note]."
Step C: Updating the GraphRAG
The summary is converted into a Vector Embedding.
This embedding is permanently attached to service [Y]'s node in the Knowledge Graph.
The Result: The next time the exact same error happens, the Triage Agent queries the payment-api node, pulls this newly created vector memory, and says: "Based on previous human feedback, we should not restart the DB, we should flush the Redis cache."
Summary of the Data Flow
Agent Input: Logs -> Graph Query: "What is the blast radius of this failing node?" -> Vector Query: "How have we fixed this node in the past?" -> Output: Highly accurate, context-aware ticket. -> Human Action: Corrects ticket -> LTM Update: New vector memory attached to the node.

What It Is
An autonomous Graph-Aware ITSM Agent — feed it a batch of JSON logs, and it uses an AI reasoning loop to produce a full incident ticket: root cause, evidence chain, recovery plan, risk score, all stored to databases and linked into a knowledge graph.

How It Works — End-to-End Flow
logs.json → main.py → LangGraph ReAct Loop → PostgreSQL + Neo4j → run_logs/INC-*.json
main.py loads data/logs.json, generates an INC-YYYY-XXXXXXXX ID, and starts streaming the LangGraph agent. Each LangGraph "node" that executes gets a Langfuse span for observability.

The LangGraph loop (graph.py) is a simple agent ↔ tools cycle:

agent node: Gemini 2.5 Flash sees the system prompt + full conversation history → decides what tool to call next.
tools node: LangChain ToolNode executes the tool, returns results → back to agent.
Loop ends when the agent produces a message with no tool calls.
The 8-Step Enforced Workflow (via SYSTEM_PROMPT)
#	Tool	What Happens
1	initialize_incident	Creates a Postgres incidents row with status ANALYZING
2a	get_service_dependencies	2-hop Cypher query → full topology map (Services, DBs, Infra, edge props like pool_size)
2b	get_blast_radius	Finds ALL services across ALL apps depending on the suspected failing resource
2c	get_infrastructure_routes	Only triggered by 502/504 gateway errors
3	search_memory	Vector similarity search in Neo4j — localized to the suspect node's linked ResolvedTicket nodes
4	add_evidence	Saves causal-chain log lines to Postgres evidence table
5	add_recovery_steps	Saves ordered fix steps to Postgres recovery_steps table
6	calculate_risk_score	Pure Python keyword detection: HIGH=+30, MEDIUM=+15, LOW=+5, capped at 100
7	finalize_incident	Updates Postgres: status → OPEN, writes plan + notes
8	save_resolved_ticket	Writes to Neo4j: MERGE a ResolvedTicket node with 768-dim Gemini embeddings (problem + solution), linked via HAS_MEMORY edges to the root cause node + all affected services
The Two Databases — Dual Purpose
PostgreSQL: Structured, queryable incident records (incidents, evidence, recovery_steps tables). The "ticket system."
Neo4j: Two jobs — (1) topology graph used for dependency/blast-radius reasoning during analysis, (2) vector long-term memory. Past incidents live as ResolvedTicket nodes linked to the services/DBs they affected.
Key Design Decisions Worth Knowing
GraphRAG, not flat vector search — Memory search is scoped to the graph node being investigated. This prevents cross-service false positives.
FastMCP bridging — Tools are defined with @mcp.tool() for MCP compatibility, then unwrapped (.fn) and re-wrapped as LangChain StructuredTools. This is why graph.py does initialize_incident.fn.
shared-analytics-db in the Neo4j graph is intentionally shared across ecommerce-prod and admin-portal — it's the designed cross-app blast-radius test case.
Human notes override AI reasoning — ResolvedTicket.human_notes is surfaced in search_memory results with a ⚠️ flag and the system prompt explicitly says those notes take priority over the agent's own analysis.
