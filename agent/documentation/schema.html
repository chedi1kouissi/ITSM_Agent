incidents Core
🔑idSERIAL PK

·app_idVARCHAR

·severityVARCHAR
·root_causeTEXT

Why this table? The central hub — every other table references it. Stores the full lifecycle of an incident from detection to resolution, including the agent's final verdict and all human-approval metadata. Without it, nothing else exists.

evidence Agent Output
🔑idSERIAL PK
🔗incident_idFK → incidents
·log_lineTEXT
·sourceVARCHAR
·timestampTIMESTAMPTZ
·reasoningTEXT
·relevance_scoreFLOAT
·log_levelVARCHAR
·categoryVARCHAR
·sequence_orderINTEGER
Why this table? Your save_relevant_evidence tool currently loses data — it just returns a string. This table is the permanent, queryable store for every log line the agent chose as proof. The sequence_order field preserves the causal chain (slow query → pool exhausted → OOMKilled).

raw_logs Input Archive
🔑idSERIAL PK
🔗incident_idFK → incidents
·app_logsTEXT
·database_logsTEXT
·infrastructure_logsTEXT
·monitoring_logsTEXT
·security_logsTEXT
·created_atTIMESTAMPTZ
Why this table? Archives the exact raw log batch that triggered analysis. Critical for audits, reproducibility, and rerunning the agent on the same input if the first analysis was wrong. You never want to lose the original evidence.

recovery_steps Remediation
🔑idSERIAL PK
🔗incident_idFK → incidents
·step_orderINTEGER
·step_descriptionTEXT
·step_typeVARCHAR
·statusVARCHAR
·risk_levelVARCHAR
·is_automatedBOOLEAN

Why this table? Breaks the recovery plan from a monolithic text blob into individually trackable, executable steps. Enables future automation (execute step → mark COMPLETED) and lets humans approve/reject individual risky steps without blocking the whole plan.

services Registry
🔑idSERIAL PK
·service_nameVARCHAR UNIQUE
·app_idVARCHAR UNIQUE
·service_typeVARCHAR
·criticalityVARCHAR
·team_ownerVARCHAR
·max_latency_msINTEGER
·max_error_rate_percentFLOAT
·db_max_connectionsINTEGER
Why this table? The agent currently hard-codes thresholds in its prompt. This table externalizes those SLO thresholds per service so the agent can query them dynamically: "Is 1400ms high for this service?" It also tells the agent who to notify and how critical a degradation really is.

incident_metrics Impact
🔑idSERIAL PK
🔗incident_idFK → incidents
·affected_users_countINTEGER
·revenue_impact_usdDECIMAL
·avg_latency_msINTEGER
·peak_latency_msINTEGER

·memory_usage_percentFLOAT

Why this table? Separates business impact data from the incident record itself. Lets you answer: "How much did this cost? How fast did we fix it?" — the numbers that matter to management. Also feeds the knowledge_base with real resolution time data.
Intelligence & Learning
🔁
similar_incidents Pattern Match
🔑idSERIAL PK
🔗incident_idFK → incidents
🔗similar_incident_idFK → incidents
·similarity_scoreFLOAT
·matching_criteriaJSONB
·created_atTIMESTAMPTZ
Why this table? A self-referencing join table that links incidents to their historical twins. When a new incident arrives, the agent can query: "Has this happened before? What fixed it?" — dramatically reducing MTTR over time. The JSONB matching_criteria stores why they're similar (same error category, same service, same log patterns).
🤖
agent_actions Audit Trail
🔑idSERIAL PK
🔗incident_idFK → incidents
·action_typeVARCHAR
·tool_nameVARCHAR
·input_paramsJSONB
·output_resultJSONB
·execution_time_msINTEGER
·successBOOLEAN
·error_messageTEXT
Why this table? Every ReAct loop step logged — tool called, what it received, what it returned, how long it took. This is non-negotiable for production AI. If the agent makes a wrong call or loops unnecessarily, you can replay and debug the exact thought chain. Also exposes cost: how many tool calls per incident?
🧠
knowledge_base Long-Term Memory
🔑idSERIAL PK
·error_patternVARCHAR
·symptom_keywordsTEXT[]
·affected_servicesVARCHAR[]
·recommended_solutionTEXT
·success_rateFLOAT
·avg_resolution_time_minutesINTEGER
·times_usedINTEGER
·created_from_incident_idVARCHAR
Why this table? The agent's long-term memory that survives between sessions. Gemini forgets everything after a run — this table doesn't. When ConnectionPoolExhausted happens for the 5th time, the agent should already know the fix. Uses PostgreSQL's GIN index on the keywords array for fast pattern matching.
👤
human_feedback HITL Loop
🔑idSERIAL PK
🔗incident_idFK → incidents
·feedback_typeVARCHAR
·feedback_textTEXT
·original_recovery_planTEXT
·modified_recovery_planTEXT
·agent_accuracy_ratingINTEGER (1–5)
·provided_byVARCHAR
·provided_atTIMESTAMPTZ
Why this table? Closes the Human-in-the-Loop cycle. When an SRE approves, rejects, or modifies a recovery plan, that feedback is gold. It captures: was the agent right? What did the human change? Over time, these diffs train better prompts and update the knowledge_base with battle-tested solutions.
